# План: пульт управления как вкладка Admin (adminTab)

Цель — «пульт на экране»: отдельная страница в левом меню Admin (как у node-red),
с живым состоянием ламп, красивым свечением реальным цветом и встроенными
сценами приложения Yeelight (свеча, закат, рассвет…).

Документ — рабочий план по фазам. Каждая фаза самодостаточна: описаны файлы,
что делать, что НЕ делать и критерии приёмки. Выполнять по порядку.

## Ключевые решения (уже приняты — не пересматривать)

1. **Пульт = вкладка Admin** (`common.adminTab` + `admin/tab_m.html`), НЕ страница
   настроек. Настройки остаются на JSON-Config как есть.
2. **Путь команд — только штатный**: вкладка делает
   `socket.emit('setState', '<base>.<KEY>', {val, ack:false})`, дальше работает
   существующий `main.onStateChange → TX_MAP`. Никаких новых каналов управления.
   `sendTo('control')` остаётся резервом, вкладка им не пользуется.
3. **UI строится по caps устройства** (те же флаги `hasCT/hasHSV/hasBG/...`, что
   гейтят TX_MAP в `core/YeelightCommands.js`). Пустые caps = показать всё
   (та же семантика, что в TX_MAP).
4. **Id объектов** — `<namespace>.<safeSegment(id || ip)>`; функция `safeSegment`
   продублирована во вкладке 1-в-1 из `core/YeelightDevice.js` (менять только синхронно!).
5. **«Встроенные сцены» ламп** (свеча, чай, закат…) по LAN не читаются из лампы —
   это пресеты приложения Yeelight, оно шлёт `start_cf`/`set_scene`. Мы воспроизводим
   их локальной библиотекой пресетов (фаза 3).
6. **Слои по AGENTS.md**: чистые данные/логика — в `core/` (dual-mode экспорт,
   без ioBroker-зависимостей), вкладка — только представление. Никаких `global.*`.

## Фаза 1 — проверка в лабе и полировка

Файлы: `admin/tab_m.html`, возможно `docs/DEBUG_RU.md`.

Задачи:
1. Задеплоить в лабу (`C:\iobroker.lab`, см. DEBUG_RU.md), убедиться что вкладка
   «Пульт Yeelight» появилась в левом меню Admin и открывается.
2. Проверить на обеих лампах (`192.168.199.100` ceilc, `192.168.199.249` color4):
   вкл/выкл, яркость, CT, hue/sat (color4), BG-блок (ceilc), Night, онлайн-точка,
   живое обновление при управлении из другого места (например, cli/control.cmd).
3. Починить найденное. Типовые риски:
   - подписка: если `socket.emit('subscribe', ...)` не работает в admin 7.8 —
     использовать `subscribeStates`;
   - `power` может приходить boolean или строкой `'on'/'off'` — хелпер `isOn()`
     уже учитывает оба, проверить фактические значения;
   - если `adapter-settings.js` мешает (тянет load/save) — заменить на прямое
     подключение `io.connect()` c query-параметрами вкладки.
4. Мелкая полировка: скролл при 5+ лампах, ширина карточек, читаемость на светлой
   теме Admin (вкладка — свой iframe, тёмный фон допустим и задуман).

Приёмка: обе лампы полностью управляются со вкладки, состояние живое (<1 с),
ошибок в консоли браузера и в логе адаптера нет.

## Фаза 2 — SVG-силуэты + живое свечение

Файлы: `admin/tab_m.html` (или вынести svg в `admin/silhouettes.js`), `core/` НЕ трогать.

Шесть силуэтов по форм-фактору (больше не надо — покрывает ~95% моделей):

| Тип | Модели (префиксы) | Особенность |
|---|---|---|
| `bulb` | mono*, color*, ct*, lamp1..9 | классическая груша E27 |
| `ceiling` | ceil*, ceiling*, ceilc | круг; если hasBG — внешнее кольцо подсветки, светится отдельно цветом bg_* |
| `strip` | strip*, stripe* | волнистая лента, градиент вдоль |
| `bedside` | bslamp* | цилиндр-башня |
| `desklamp` | desklamp, lamp15?, mila? | лампа на ноге/дуге |
| `screenbar` | monitor*, screen* | горизонтальный бар |

Правила:
- Маппинг `model → тип` — таблица префиксов + фолбэк по caps
  (hasBG → ceiling, hasHSV без BG → bulb, иначе bulb).
- Свечение = заливка/фильтр силуэта реальным цветом (функции `lampColor`,
  `kelvinToRgb`, `hslCss` уже есть в tab_m.html — переиспользовать) +
  `drop-shadow`, интенсивность от яркости. Выключена — тёмно-серый контур.
- У ceiling с BG: основной диск светится основным светом, кольцо — bg-цветом.
- Не рисовать фотореализм по конкретным моделям — решение принято, эффект
  делает живой цвет, а не детализация.

Приёмка: ceilc отображается «люстрой с кольцом» (кольцо живёт своей жизнью от
bg_*), color4 — лампочкой; цвет/яркость свечения меняются в реальном времени.

## Фаза 3 — сцены-пресеты («как в приложении»)

Файлы: НОВЫЙ `core/YeelightScenes.js` (чистый, dual-mode), `core/YeelightCommands.js`,
`admin/tab_m.html`.

1. `core/YeelightScenes.js` — библиотека пресетов. Каждый пресет:
   `{ key, label_ru, label_en, icon, needs: 'ct'|'color', build() }`, где `build()`
   возвращает либо `{ method:'start_cf', params:[count, action, expr] }`, либо
   `{ method:'set_scene', params:[...] }`. Минимальный набор:
   - `sunset` Закат — cf: тёплый оранжевый → красный → выкл (~15 мин классика,
     сделать параметр длительности, по умолчанию короче);
   - `sunrise` Рассвет — обратный;
   - `candle` Свеча — бесконечный cf, мерцание ~2700-2900K / низкая яркость;
   - `romance` Романтика — медленный цикл розовый/фиолетовый (needs color);
   - `night` Ночник — set_scene nightlight / минимальная яркость тёплого;
   - `movie` Кино — приглушённый тёплый статичный;
   - `tea` Чаепитие — статичный ~3000K средней яркости.
   Таблицы значений брать из известных пресетов Yeelight-приложения (они же в
   python-библиотеке yeelight, `flows.py` — сверить числа там).
2. В `buildTxMap` (гейт `hasScene` ИЛИ `hasFlow`): состояние `scene_preset`
   (string) — принимает `key` пресета, находит его в YeelightScenes, шлёт команду.
   Неизвестный key → warn, null. Пустая строка/`'0'` → `stop_cf`.
3. Во вкладке — ряд чипов-сцен (иконка + название) на карточке, фильтр по
   `needs` против caps (color-сцены не показывать CT-only лампе). Активную сцену
   не подсвечивать (лампа не сообщает её) — чипы это кнопки, не индикаторы.
4. JSDoc на публичные функции YeelightScenes (правило AGENTS.md).

Приёмка: «Свеча» на ceilc даёт видимое мерцание; «Закат» на color4 плавно гаснет;
`stop_cf`/OFF корректно прерывают; сцены доступны и из ioBroker-объектов
(setState `scene_preset` = 'candle' руками).
